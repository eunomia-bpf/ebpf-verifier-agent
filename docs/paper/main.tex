\begin{abstract}
eBPF underpins critical infrastructure in modern Linux systems, yet when its
static verifier rejects a program, developers must decipher verbose abstract
interpretation logs spanning hundreds of lines. Existing diagnostic methods
either apply regular expressions to the final error line or feed unstructured
text to language models, neither reliably identifying root-cause proof
failures. Our analysis of 591 production verifier-fix commits finds that
64\% are proof-reshaping workarounds stemming from systematic
misdiagnosis.

We observe that the verifier's \texttt{LOG\_LEVEL2} output is already the
complete execution trace of an abstract interpreter---it contains where safety
proofs are established, propagated, and lost, but this structure is buried in
flat text. OBLIGE performs meta-analysis on this trace: it infers proof
obligations from the verifier's type system, tracks them as formal predicates
across every instruction, and identifies the precise transition point where a
proof breaks. Evaluated on 302 real failures from kernel selftests, Stack
Overflow, and production projects, OBLIGE achieves 94\% proof obligation
coverage with median latency of 27\,ms, and improves LLM-assisted repair
accuracy on the hardest failure class by 30 percentage points over raw
verifier output.
\end{abstract}

%% ==============================
\section{Introduction}
%% ==============================

eBPF has become critical infrastructure in modern Linux deployments.  Major
production systems---Cilium, Meta's Katran, the Linux kernel's own
\texttt{bpftrace} and \texttt{bcc} frameworks, and cloud-provider networking
stacks---rely on eBPF programs for packet processing, security policy
enforcement, performance tracing, and system call interposition.  The Linux
eBPF verifier is the gatekeeper: every program must pass its safety checks
before being allowed to run in the kernel.

\subsection{The Diagnostic Gap}
\label{sec:intro-gap}

When the verifier rejects a program, it emits a verbose log at
\texttt{LOG\_LEVEL2}.  This log is a \emph{complete execution trace of the
verifier's abstract interpreter}: for each of the program's BPF instructions,
it records the full abstract state of all registers (type, scalar bounds,
pointer offset, packet range, BTF provenance), control flow merge points, and
the verifier's own backward precision-tracking annotations
(\texttt{mark\_precise}).  A typical rejection trace spans 500 to 1000 lines;
production cases can exceed 4000 lines~\cite{deokar2024}.

Despite containing all the information needed to understand the failure, this
trace is effectively unreadable.  The final error line names a \emph{symptom}:
``\texttt{math between pkt pointer and register with unbounded min value is not
allowed}.''  But the \emph{cause}---which instruction destroyed the scalar
bounds proof, and why---is buried 30 to 500 lines earlier.  An empirical study
of 743 Stack Overflow questions finds that 19.3\% are verifier-related, with
debugging and error-message comprehension as the dominant
difficulties~\cite{deokar2024}.  The Linux Foundation's security audit of the
eBPF verifier independently recommends improving documentation of ``what the
verifier enforces and why''~\cite{nccaudit2024}.

Three concrete manifestations of this diagnostic gap stand out in our corpus
of 302 real verifier failures:

\begin{enumerate}
  \item \textbf{Information is present but buried.}  The verifier already emits
        complete per-instruction abstract state at \texttt{LOG\_LEVEL2}.  The
        information to understand the failure is \emph{in} the trace---but
        it is a needle in a 500-line haystack.

  \item \textbf{Existing tools ignore the trace.}  Pretty Verifier~\cite{prettyverifier2025}
        applies 91 regular expressions to the \emph{final error line} only.
        LLM-based tools such as Kgent~\cite{kgent2024} and SimpleBPF~\cite{simplebpf2025}
        feed the full raw log as unstructured text to language models, which
        cannot reliably extract state transitions from prose.

  \item \textbf{Developers systematically misdiagnose lowering artifacts.}
        When LLVM lowering destroys a proof that exists at the source level,
        the error message points at the symptom (the final failing instruction)
        and developers add bounds checks at the wrong location.  Our analysis
        of 591 production verifier-fix commits across Cilium, Katran, bpftrace,
        bcc, and Aya finds that 63.6\% are \emph{proof-reshaping workarounds}---
        source changes that restructure code so the verifier can re-derive a
        proof it lost during lowering---not genuine safety bug fixes.
\end{enumerate}

\subsection{The Key Insight}

The eBPF verifier \emph{is} a static abstract interpreter~\cite{cousot1977}.
Its \texttt{LOG\_LEVEL2} output is the complete execution trace of that abstract
interpreter.  Every register's type, bounds, offset, and range at every
instruction; every control flow merge; every precision-tracking backtrack---all
are present in the log.

The missing step is \emph{meta-analysis}: treating the verifier's abstract
interpretation output as a first-class data structure and reasoning about it.
For a given failure, what proof obligation did the verifier need?  At which
instruction was that obligation satisfied?  At which instruction did it become
violated?  What caused the transition?  Does the source code contain a correct
proof that LLVM's lowering destroyed?

Rust's borrow checker provides the right analogy.  When Rust cannot prove
memory safety, it does not emit a single error line pointing at the failed
dereference; it shows multiple source locations with causal labels
(``borrow occurs here,'' ``use occurs here,'' ``conflict here'').  OBLIGE
does the same for eBPF safety proofs, with labels such as
\emph{proof-established}, \emph{proof-lost}, and \emph{rejected}.

\texttt{bpf\_map\_lookup\_elem()} and immediately dereferencing the result

finds \texttt{proof\_status = never\_established}: the register type is
\texttt{map\_value\_or\_null} from instruction~0 through to rejection, with
no point where the obligation was ever satisfied.  The developer sees
\emph{proof never established} (rather than \emph{proof established then
lost}), immediately recognizing a genuine source bug rather than a compiler
issue.

\subsection{Contributions}

\begin{itemize}
  \item \textbf{Proof trace meta-analysis.}  A technique for second-order
        analysis of abstract interpretation output.  OBLIGE infers formal proof
        obligations from eighteen obligation families (e.g., \texttt{off+sz<=range}
        for packet access), evaluates them as boolean predicates
        at every instruction in the trace, and identifies the \emph{transition
        witness}---the exact instruction where a proof predicate changes from
        satisfied to violated.

  \item \textbf{OBLIGE diagnostic engine.}  A pure-userspace tool implementing
        the full pipeline: log parsing, state trace parsing (including
        \texttt{mark\_precise} backtracking extraction), proof obligation
        inference, predicate tracking with register value lineage, BTF source
        correlation, and Rust-style multi-span diagnostic rendering.
        Median latency 27\,ms; no kernel patches required.  120 unit tests
        cover obligation inference, predicate evaluation, and transition
        witness detection.

  \item \textbf{Empirical findings on 302 real failures.}  (a) OBLIGE
        generates diagnostics for 241/241 cases with 0 crashes, 94.2\%

        confirms 63.6\% are proof-reshaping workarounds.
\end{itemize}

\section{Background}

\subsection{The eBPF Verification Pipeline}

eBPF programs are written in C or Rust, compiled by LLVM to BPF bytecode, and
submitted to the kernel via the \texttt{bpf()} syscall.  The Linux verifier
runs before any program executes.  It performs a dataflow-style abstract
interpretation~\cite{cousot1977,gershuni2019simple}: it walks all execution
paths through the bytecode, maintaining per-register abstract state and
enforcing a set of safety properties at each instruction.  Programs that
violate any property are rejected.

The key architectural point is that there are three distinct reasoning levels:
\begin{itemize}
  \item \textbf{Source level.}  The programmer reasons about C/Rust values,
        bounds checks, null checks, and pointer lifetimes.
  \item \textbf{Compiler level.}  LLVM transforms source into BPF bytecode,
        potentially splitting, merging, or reordering computations.
  \item \textbf{Verifier level.}  The verifier reasons about BPF register
        abstract state: types, scalar bounds (\texttt{umin/umax},
        \texttt{smin/smax}), pointer offsets, packet ranges, and provenance IDs.
\end{itemize}
A proof that is valid at the source level can be lost at the verifier level if
LLVM's lowering produces register state that the verifier's abstract domain
cannot represent.  This is the root cause of lowering artifacts.

\subsection{Anatomy of a \texttt{LOG\_LEVEL2} Trace}
\label{sec:trace-anatomy}

When \texttt{log\_level=2} is set in \texttt{BPF\_PROG\_LOAD}, the verifier
emits a complete per-instruction trace interleaving per-register abstract state
(type, scalar bounds, pointer offset, packet range, provenance ID), BTF source
annotations~\cite{linuxbtf}, \texttt{mark\_precise} backtracking chains, and
control flow merge points.  A typical 50-instruction program produces
100--300 lines.  We detail the parsing of each element in~\S\ref{sec:state-trace-parsing}.

\subsection{State Trace Parsing}
\label{sec:state-trace-parsing}

The log parser (Stage 1) handles error message identification: it matches
the final error line against 23 error patterns (OBLIGE-E001 through E023),
covering 87.1\% of failures in our 302-case corpus.  Each pattern maps to
a taxonomy class (\textit{source\_bug}, \textit{lowering\_artifact},
\textit{verifier\_limit}, \textit{env\_mismatch}, \textit{verifier\_bug}).

The state trace parser (Stage 2) parses the remaining structure.  For each
BPF instruction, it extracts:
\begin{itemize}
  \item Instruction index, opcode, and mnemonic
  \item Pre/post-state per live register (type, \texttt{umin/umax},
        \texttt{smin/smax}, offset, packet range, provenance ID)
  \item BTF source annotation (file, line, column) if present
  \item Backtracking links from \texttt{mark\_precise} annotations
  \item Branch merge points with state at each join
\end{itemize}

The primary challenge is format variability.  Register state can appear before
or after the instruction, in branch merge lines, in backtracking annotations,
or across multiple lines.  The parser normalizes all variants into a uniform
\texttt{TracedInstruction} representation.

\textbf{Backtracking chain extraction.}  The verifier's own
\texttt{mark\_precise} backward pass is emitted in the log as
\texttt{regs=R stack=S before K: (op)~...} annotations.  OBLIGE extracts
these into a \texttt{BacktrackChain} structure: a sequence of
(instruction\_index, registers\_tracked) pairs.  This is the verifier's own
root-cause chain, expressed as debug text; OBLIGE is the first tool to
extract and structure it.

\subsection{Proof Obligation Inference}
\label{sec:obligation-inference}

Given the error message and the register state at the rejection point, OBLIGE
infers what the verifier \emph{needed} to see: the \emph{proof obligation}.
This is expressed as a formal predicate over register state.

OBLIGE supports eighteen obligation families, shown in Table~\ref{tab:obligations}.
The error message identifies the obligation \emph{family}.  The register state
at the rejection point provides concrete parameters.  OBLIGE instantiates the
predicate template with these parameters.

This predicate instantiation is not pattern matching on error text.  The same
error message (\texttt{invalid access to packet}) with different register
states at the rejection point produces different predicate instances with
different parameters.  The predicate is grounded in the verifier's type system:
for packet access, the verifier literally checks
\texttt{reg.off~+~access\_size~<=~reg.range}; OBLIGE infers this from the
register fields at the rejection point.

\textbf{Inference algorithm.}  For each error ID, OBLIGE defines:
(1) which register(s) contain the failing object (extracted from the rejection
line), (2) which predicate template applies (from the obligation family), and
(3) how to extract concrete parameters from the register state dump.  The
resulting predicate is a boolean function over register state that can be
evaluated at any point in the trace.

\subsection{Formal Predicate Tracking}

Given the instantiated predicate $P$ from Section~\ref{sec:obligation-inference},
OBLIGE evaluates $P$ at every instruction in the trace.  This produces a boolean
sequence over the trace:
\[
  \underbrace{?, \ldots, ?}_{\text{unknown}},
  \underbrace{\top, \ldots, \top}_{\text{satisfied}},
  \underbrace{\bot, \ldots, \bot}_{\text{violated}}
\]

OBLIGE assigns proof lifecycle labels to each instruction:
\begin{itemize}
  \item \emph{proof\_established}: the first instruction where $P$ becomes
        satisfied (\texttt{?} $\to$ \texttt{true})
  \item \emph{proof\_holds}: instructions where $P$ remains satisfied
  \item \emph{proof\_lost}: the first instruction where $P$ transitions from
        satisfied to violated (\texttt{true} $\to$ \texttt{false})---the
        \emph{transition witness}
  \item \emph{rejected}: the final error instruction (always identified)
\end{itemize}

If $P$ is never satisfied, the status is \textit{never\_established} (a source
bug: the proof obligation was never met, indicating a missing safety check).
If $P$ is satisfied but becomes violated before the rejection, the status is
\textit{established\_then\_lost} (a lowering artifact or proof propagation
failure: the source has a valid check but the lowered code loses it).

\textbf{Key distinction from heuristic approaches.}  The transition witness is
not ``detect a bounds collapse pattern.''  It is ``evaluate the specific
predicate that the verifier needed at this specific rejection point, and find
the exact instruction where it broke.''  Different predicates produce different
transition witnesses on the same trace.  This is formally grounded in the
verifier's type system, not in ad hoc pattern detection.

\textbf{Register value lineage.}  Predicates reference registers, but register
names are reused across the trace.  A value computed in R0 may be copied to R3
before the access.  OBLIGE tracks value identity across register moves and
copies.  If R3 is defined as a copy of R0 (\texttt{r3 = r0}), and the
obligation predicate references the packet range of R0, OBLIGE continues
tracking the range on R3 for subsequent instructions.  This register value
lineage is essential for correctly identifying proof-established and proof-lost
events across LLVM's register allocation.

\textbf{Proof propagation analysis.}  For lowering artifact detection, OBLIGE
additionally checks whether the proof established at the bounds-check site
\emph{propagates} to the register used at the access site.  If a different
register is used for the access than the one whose range was established at
the check (a common LLVM pattern), OBLIGE classifies this as a lowering
artifact regardless of whether the proof predicate was technically satisfied
at some earlier instruction.

\subsection{Formal Foundation: Meta-Analysis of Abstract Interpretation Output}
\label{sec:formal-foundation}

The predicate tracking described above can be understood as a
\emph{second-order abstract interpretation}: OBLIGE applies abstract
interpretation to the \emph{output} of another abstract
interpreter~\cite{cousot1977}.  This section formalizes the framework.

\subsubsection*{Two layers of abstract interpretation}

\textbf{Layer 1} is the eBPF verifier itself.  Its input is BPF
bytecode; its abstract domain $D_1$ consists of register types, scalar
bounds $[\texttt{umin},\texttt{umax}]$, pointer offsets, and packet ranges.
Its output is a trace $T = (s_0, s_1, \ldots, s_n)$ where each $s_i \in D_1$
is the abstract state after the $i$-th instruction.

\textbf{Layer 2} is OBLIGE.  Its input is the trace $T$; its abstract domain
$D_2$ tracks proof obligation status; its output is a sequence of proof
lifecycle labels over $T$.

\begin{definition}[Obligation status lattice]
\label{def:lattice}
The obligation status lattice is
$\mathcal{L} = \{\bot, \mathit{unknown}, \mathit{satisfied}, \mathit{violated}\}$
with ordering
$\bot \sqsubseteq \mathit{unknown} \sqsubseteq \mathit{satisfied}$
and
$\bot \sqsubseteq \mathit{unknown} \sqsubseteq \mathit{violated}$,
where $\mathit{satisfied}$ and $\mathit{violated}$ are incomparable.
The bottom element~$\bot$ represents an uninitialized status
(obligation predicate not yet evaluable); $\mathit{unknown}$ represents a
state where the predicate is evaluable but the relevant registers are not
yet constrained.
\end{definition}

\begin{definition}[Obligation transfer function]
\label{def:transfer}
Let $P : D_1 \to \{\mathit{true}, \mathit{false}, \mathit{unknown}\}$ be the
obligation predicate instantiated from the rejection point
(\S\ref{sec:obligation-inference}).
For each traced instruction~$i$ with verifier state $s_i \in D_1$,
the obligation transfer function $\tau_i : \mathcal{L} \to \mathcal{L}$ is:
\[
\tau_i(\ell) = \begin{cases}
  \mathit{sat} & P(s_i) \land \ell \in \{\mathit{unk}, \mathit{sat}\} \\[2pt]
  \mathit{vio} & \neg P(s_i) \land \ell = \mathit{sat} \\[2pt]
  \ell         & \text{otherwise}
\end{cases}
\]
\end{definition}

\noindent
The status sequence is computed by forward iteration:
$\ell_0 = \mathit{unknown}$,\;
$\ell_i = \tau_i(\ell_{i-1})$.

\begin{definition}[Transition witness]
\label{def:witness}
The transition witness is the smallest index $w$ such that
$\ell_{w-1} = \mathit{satisfied}$ and $\tau_w(\ell_{w-1}) = \mathit{violated}$.
This is the exact instruction where the proof breaks.
If no such $w$ exists and $\ell_n = \mathit{violated}$, the proof was
\emph{never established}.
\end{definition}

\begin{proposition}[Soundness of lifecycle labels]
\label{prop:soundness}
If OBLIGE labels instruction~$i$ with
$\ell_i = \mathit{satisfied}$, then for all concrete states
$c \in \gamma(s_i)$ (where $\gamma$ is the concretization function of the
verifier's abstract domain), the safety property encoded by~$P$ holds.
\end{proposition}
\begin{proof}[Proof sketch]
The verifier is a sound abstract interpreter: $s_i$ over-approximates all
concrete states reachable at instruction~$i$~\cite{gershuni2019simple}.
The obligation predicate $P$ is monotone with respect to the verifier's
partial order on~$D_1$ (tighter bounds imply satisfaction if looser bounds
already satisfy).  Therefore $P(s_i) = \mathit{true}$ implies
$P(c) = \mathit{true}$ for all $c \in \gamma(s_i)$.
\end{proof}

\textbf{Backward obligation slice.}
Given the transition witness $w$, the \emph{backward obligation slice} is
\[
  S = \{ i < w \mid \exists\, r \in \mathit{regs}(P) .\;
    \text{instruction } i \text{ writes to } r \}
\]
i.e., all instructions before $w$ that write to registers appearing in the
obligation predicate~$P$.  When \texttt{mark\_precise} backtracking
information is available in the trace, $S$ is refined to include only
instructions on the backtracking chain, yielding a tighter slice.

\textbf{Generalization beyond eBPF.}
This framework applies to any abstract interpreter that outputs per-step
abstract states.  The requirements are: (1)~per-step abstract states are
emitted, (2)~safety properties are expressible as predicates over the
abstract domain, and (3)~the trace is ordered.  Rust's borrow checker
(lifetime predicates over borrow state), WebAssembly validators (type
predicates over operand stack state), and Java bytecode verifiers (type
predicates over stack frame state) all satisfy these
requirements---suggesting that second-order trace analysis may be a general
technique for improving diagnostic quality in verified language runtimes.

\subsection{BTF Source Correlation}

OBLIGE maps each proof lifecycle event (established, lost, rejected) to its
source location via BTF \texttt{line\_info} annotations~\cite{linuxbtf}.
When the verifier emits a source-location comment before an instruction (e.g.,
\texttt{; foo.c:42}), OBLIGE records the mapping.  Multiple consecutive bytecode
instructions from the same source line are merged into a single span.

The result is 3--5 source-level spans, each labeled with a proof lifecycle
role and annotated with the register state change at that point.  When BTF
annotations are absent (37.3\% of our corpus, primarily Stack Overflow cases
where log fragments lack BTF context), OBLIGE falls back to bytecode-level
spans (instruction indices only).

BTF coverage is a data quality issue, not a parser limitation: kernel
selftest cases have 98.7\% BTF coverage, confirming that the correlation works
correctly when the input has BTF annotations.

\subsection{Diagnostic Rendering}

OBLIGE produces two output formats from the same proof lifecycle data:

\textbf{Human-readable Rust-style output.}  Modeled on Rust compiler
diagnostics~\cite{rustdiag2016}, the output shows multiple source spans
with causal labels, register state changes at each transition, and
\texttt{note}/\texttt{help} annotations.  The format is designed to be
readable in a terminal, an IDE, or a CI log without any tool support.

\textbf{Structured JSON.}  A machine-readable format containing
\texttt{error\_id}, \texttt{taxonomy\_class}, \texttt{proof\_status}, an array
of \texttt{spans} (each with role, source location, instruction index, and
state change), the instantiated obligation predicate, and \texttt{note}/\texttt{help}
text.  This format is designed for consumption by LLM agents (replacing raw
log text in prompts), CI systems (structured failure annotations), and IDE
plugins.

\subsection{Implementation Notes}

OBLIGE is implemented in Python ($\sim$6,600 lines across five modules:
\texttt{log\_parser}, \texttt{trace\_parser}, \texttt{proof\_engine},
\texttt{source\_correlator}, and \texttt{rust\_diagnostic}), with 120 unit
tests covering obligation inference, predicate evaluation, backward slicing,
and transition witness detection on hand-crafted and real-corpus traces.
It defines 23 error IDs (OBLIGE-E001 through E023) and eighteen obligation
families.

\textbf{Source-code independence.}  OBLIGE requires only the verifier log
text as input---no object files, source files, or external debug information.
When BTF \texttt{line\_info} annotations are present in the log, OBLIGE maps
proof events to source-level spans; when BTF is absent, it falls back to
bytecode-indexed spans.  For already-loaded programs, \texttt{bpftool prog
dump xlated linum} can supply supplementary source context, but this is
optional.

\textbf{Language independence.}  Because OBLIGE analyzes the verifier trace
at the BPF bytecode level, it is agnostic to the source language.  We
validated the pipeline on 18 Rust (Aya) and 7 Go (Cilium) cases alongside the
C corpus: diagnostic generation succeeds on all 25 cases, with obligation
inference on 17/25 (68\%) and taxonomy classification on 25/25 (100\%).
The lower obligation rate for non-C cases reflects thinner logs in the stored
corpus, not a language-specific limitation.

The headline overall fix-type accuracy shows Condition A marginally ahead
(85.2\% vs 79.6\%), but the per-class breakdown tells the story.  On lowering
artifacts---the class where raw error messages are most misleading---OBLIGE
diagnostics double the fix-type accuracy: 3/10 (30\%) with raw logs versus
6/10 (60\%) with OBLIGE diagnostics, a +30\,pp gain.  In Condition A, the LLM

the correct repair.  The raw log for these cases already named the specific
contract violation, and the generic OBLIGE output added noise.  This is a
known issue with BTF-absent traces and is targeted for improvement.

We present this as evidence that OBLIGE's value is concentrated on the
lowering-artifact class, where proof-trace analysis provides qualitatively
different information not available in raw logs.

The key distinction is not classification accuracy.  It is root-cause

has only a 7-instruction snippet, and one requires cross-subprogram analysis
not yet implemented).

via \texttt{llvm\allowbreak-objdump} on the compiled object file (unavailable
in most deployment contexts).  OBLIGE uses BTF annotations embedded in the log
itself, achieving 62.7\% coverage without any external files.

\subsection{Runtime Overhead (Q6)}
\label{sec:latency}

OBLIGE processes all 241 cases with median latency 27\,ms (P95: 43\,ms,
max: 92\,ms, scaling linearly with log length at Pearson $r = 0.74$).
All cases complete in under 100\,ms---comfortably within interactive bounds
and negligible compared to a single LLM API call (1--10\,s).

\section{Discussion and Limitations}
\label{sec:discussion}

\subsection{Scope and Honest Limitations}

\textbf{94\% obligation coverage.}  Eighteen obligation families cover 94.2\% of
cases.  The remaining 6\% involve environment errors (Permission denied, build
failures) rather than true verifier rejections; these fall outside the scope of
proof-trace analysis.  The predicate framework is extensible, and adding a new
family requires defining the predicate template and the parameter extraction
rules.  We provide an explicit enumeration of the eighteen supported families
(Table~\ref{tab:obligations}) so readers can assess scope.

\textbf{BTF coverage.}  37.3\% of cases lack BTF annotations (primarily Stack
Overflow fragments).  Without BTF, OBLIGE produces bytecode-level spans
(instruction indices) rather than source-level spans.  This degrades human
readability but not the underlying predicate analysis.  The 98.7\% BTF coverage
on kernel selftests confirms that the correlation is correct when inputs are
well-formed.

\textbf{Path-insensitive slicing.}  OBLIGE's backward slice from the
transition witness is bounded and path-insensitive.  It will not correctly
handle all loop- or alias-sensitive proof loss patterns.  We scope the slice
as an explanatory aid after the main predicate/transition detection, and
validate it empirically rather than claiming formal precision.

\textbf{LLM repair experiment uses proxy metrics.}  The A/B experiment uses 54
cases and proxy metrics (fix-type classification) rather than verifier-pass
outcomes.  The regression on source-bug cases caused by BTF-absent traces
pointing toward metadata issues is a known failure mode.  A full re-run with
updated prompts and a verifier-pass oracle is planned.

\textbf{No cross-kernel stability evaluation.}  The verifier's log format
varies across kernel versions.  We have not yet evaluated OBLIGE across
multiple kernel versions.  Prior work~\cite{depsurf2025} finds that 83\% of
eBPF extensions are affected by at least one kernel mismatch; this is an
important axis for future work.

\subsection{Upstream Potential}

OBLIGE's output could become part of the kernel's own diagnostic
infrastructure.  The structured JSON format and 23 error IDs are designed
to be stable and extensible.  The proof obligation catalog could inform
improvements to verifier error messages.  The Rust-style rendering could
be integrated into \texttt{bpftool} or the \texttt{libbpf} loader, so
developers receive structured diagnostics directly from the load path.

\section{Related Work}
%% ==============================

\subsection{eBPF Verifier Complexity and Developer Pain}

Jia et al.~\cite{hotos2023verif} argue that kernel-extension verification
is ``untenable'' as the extension surface grows; OBLIGE addresses the near-term
reality that millions of C/Rust eBPF programs run against the current verifier.
Rex~\cite{rex2025} studies 72 verifier workaround commits, confirming 44.4\% are
LLVM code-generation changes; Rex explains \emph{why} such failures exist, OBLIGE
explains \emph{this specific failure instance} at load time.  Deokar et
al.~\cite{deokar2024} find 19.3\% of 743 Stack Overflow questions are
verifier-related, quantifying the pain point OBLIGE addresses.

\subsection{Diagnostic Tools}

Pretty Verifier~\cite{prettyverifier2025} applies 91 regex patterns to the final
verifier error line, producing a single-span explanation; it cannot detect
proof-loss transitions and crashes on 10.7\% of our corpus.  OBLIGE processes
the complete state trace with formal predicate evaluation---a different level of
analysis that enables root-cause localization and lowering artifact diagnosis.
The \texttt{ebpf-verifier-errors} repository~\cite{ebpfverifdict} crowdsources
(log, fix) pairs as data, but provides no automated analysis.

\subsection{LLM-Based eBPF Tools}

Kgent~\cite{kgent2024} feeds raw verifier output verbatim to an LLM repair loop;
OBLIGE's structured diagnostics could serve as Kgent's preprocessor, replacing
unstructured register dumps with labeled causal spans.  SimpleBPF~\cite{simplebpf2025}
generates a high-level DSL and explicitly notes that raw-log retry ``was not effective'';
OBLIGE addresses the general case (arbitrary handwritten C/Rust) that DSLs cannot
replace.  K2~\cite{k22023} optimizes eBPF programs for performance, a goal
orthogonal to failure diagnosis.

\subsection{Static Analysis and Diagnostics}

OBLIGE's approach is inspired by counterexample analysis in model checking:
reconstruct why verification failed from the evidence trace.  Abstract
interpretation~\cite{cousot1977} and its formalization for eBPF~\cite{gershuni2019simple}
ground OBLIGE's obligation predicates.  Program slicing~\cite{weiser1981} motivates
backward dependency extraction from the transition witness.  Rust's compiler
diagnostics~\cite{rustdiag2016} pioneered multi-span source-level errors with
causal labels; research on compiler error messages~\cite{errormessages2014} confirms
that pointing at the wrong location significantly harms developer productivity.

%% ==============================
\section{Conclusion}
%% ==============================

The eBPF verifier already emits---in its \texttt{LOG\_LEVEL2} trace---everything
needed to understand why a program is rejected.  OBLIGE demonstrates that
meta-analysis of this abstract interpretation output enables Rust-quality
multi-span diagnostics: formal predicate tracking to identify exactly where
proof obligations are established, propagated, and lost.  The clearest payoff
is for lowering artifacts, where line-oriented tools localize root causes in
0\% of cases; OBLIGE correctly diagnoses 67\% and doubles LLM repair accuracy,
enabling correct repairs that the raw log cannot guide.  As eBPF programs grow in complexity, structured proof trace
analysis at the scale and latency demonstrated here (27\,ms, no kernel patches)
is a practical path to making verifier failures comprehensible.
